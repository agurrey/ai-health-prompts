  <name>Task 1: Create sync.ts offline-first sync engine</name>
  <files>app/src/lib/sync.ts</files>
  <action>
Create `app/src/lib/sync.ts` — the offline-first sync engine. This module handles all Supabase DB writes with queueing and retry logic.

**Sync Queue (localStorage-backed):**
- Key: `hormesis_sync_queue`
- Format: `SyncQueueItem[]` from types/community.ts
- Helper functions: `loadQueue(): SyncQueueItem[]`, `saveQueue(items: SyncQueueItem[]): void`, `addToQueue(item: Omit<SyncQueueItem, 'id' | 'created_at' | 'retries'>): void`
- Generate IDs with `crypto.randomUUID()` (available in all modern browsers)

**Core sync functions:**

1. `syncWorkout(workout: CompletedWorkout, xpEarned: number)` — Upserts to `workouts` table. On failure, adds to queue. Non-blocking (fire-and-forget with `.catch(queueFallback)`).

2. `syncExerciseLogs(entries: ExerciseLogEntry[])` — Upserts to `exercise_logs` table. Batch operation. On failure, adds individual items to queue.

3. `syncAchievement(achievementId: string, unlockedAt: string)` — Inserts to `user_achievements`. On failure, queues.

4. `postFeedEvent(eventType: string, eventData: Record<string, string | number>)` — Inserts to `activity_feed`. Only if user has a public profile (check profile.is_public). CRITICAL: eventData must NEVER contain weight or time values. The caller is responsible for sanitizing, but this function should also strip any key containing 'weight', 'time', 'kg', 'lbs' as a safety net.

5. `drainSyncQueue()` — Processes all items in the queue. For each item: attempt the Supabase operation. On success: remove from queue. On failure: increment retries. If retries > 5: discard the item (data is still in localStorage). Call this on app startup if user is authenticated, and after reconnect.

6. `getSyncStatus()` — Returns `{ queueLength: number; lastSyncAt: string | null; connectionState: ConnectionState }`.

**Auto-pause handling:**
- Before any Supabase operation, call `pingSupabase()` first.
- If ping returns false (paused), don't attempt the operation — add to queue instead.
- The UI layer (Plan 05) will show "Connecting..." based on ConnectionState.
- When state changes from 'paused' to 'connected', auto-drain the queue.

**Last-write-wins strategy:**
- Use Supabase `.upsert()` with `onConflict` matching the unique constraint.
- For workouts: conflict on (user_id, date).
- For exercise_logs: conflict on (user_id, exercise_id, date).
- For user_achievements: conflict on (user_id, achievement_id).
- The most recent write (by synced_at timestamp) wins.

**Helper: `isCommunityUser()`** — Checks if there's an active auth session. If not, all sync functions return immediately without doing anything. This ensures zero overhead for users who haven't joined.

Import `getSupabase, pingSupabase, getAuthUser, getConnectionState, onConnectionStateChange` from supabase.ts. Import types from types/community.ts. Import `CompletedWorkout, ExerciseLogEntry` from storage.ts (use `import type` to avoid circular deps).

Register a connection state listener at module level that auto-drains queue when state transitions to 'connected':
```typescript
if (typeof window !== 'undefined') {
  onConnectionStateChange((state) => {
    if (state === 'connected') drainSyncQueue();
  });
}
```
